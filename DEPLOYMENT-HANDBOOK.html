Enterprise logistics & delivery platform — build, run, and operate the MVP stack.

Monorepo · pnpm + Turborepo Next.js 14 · NestJS 10 PostgreSQL 16 · Redis 7 CI: lint · typecheck · test · build MVP slice — not full platform

1. Project Overview

Livraison is a multi-tenant logistics platform. This handbook covers the implemented MVP slice: three backend services, a PostgreSQL data layer, and three web portals, plus full container, CI/CD, and single-host deployment automation.

Implemented components

ComponentTypePortPurpose
identityNestJS service3001Auth (JWT access+refresh), users, tenants, RBAC
shipmentNestJS service3002Shipment lifecycle, status workflow, tracking numbers
trackingNestJS service3003Tracking events, timeline, public tracking, SSE, audit
core-dbPrisma data layer—Schema, migrations, Row-Level Security, seeds
web-merchantNext.js app3100Merchant portal (ship, track, settle views)
web-adminNext.js app3101Admin portal (users, tenants, shipments, audit)
web-driverNext.js PWA3102Driver portal (offline-first deliveries, POD, GPS)
Scope honesty: the blueprint describes a FedEx-class platform (~28 services). This MVP implements the core auth → shipment → tracking loop and three portals (~20–25% of the documented scope). Present it as a working MVP with a roadmap, not the complete platform.

2. Architecture Overview

Domain-aligned services with their own PostgreSQL databases. Web apps never talk to services directly from the browser — each app has a server-side Backend-for-Frontend (BFF) that holds JWTs in an httpOnly cookie and injects them into upstream calls.

Browser
  │  (httpOnly cookie session; no tokens in JS)
  ▼
Next.js app  ──BFF route handlers──▶  identity (3001)  ─▶  PostgreSQL: identity
  (3100/3101/3102)                    shipment (3002)  ─▶  PostgreSQL: shipment
                                      tracking (3003)  ─▶  PostgreSQL: tracking
                                         │
                                      Redis (sessions, revocation, idempotency)

Nginx (host) terminates TLS and reverse-proxies the three apps + public API.
Postgres & Redis are NOT exposed to the network (internal only).

Key patterns

  • Multi-tenancy: every row carries tenant_id; PostgreSQL Row-Level Security enforces isolation and fails closed when no tenant context is set.
  • Auth: Argon2id passwords; short-lived JWT access token + rotating refresh token with reuse detection.
  • Errors: RFC 7807 application/problem+json with stable codes and request IDs.
  • Realtime: Server-Sent Events for live tracking updates.
  • Offline (driver): IndexedDB durable outbox with idempotency keys + background sync.

5. Local Development Setup

Prerequisites (all OSes)

Node.js 20.11+, pnpm 9 (via Corepack), Docker Desktop / Docker Engine, Git.

Windows

# PowerShell
corepack enable
corepack prepare pnpm@9.12.3 --activate
git clone <repo> C:\livraison; cd C:\livraison
pnpm install
# start datastores only
docker compose up -d postgres redis
# generate Prisma clients
pnpm --filter @livraison/identity exec prisma generate
pnpm --filter @livraison/shipment exec prisma generate
pnpm --filter @livraison/tracking exec prisma generate
# run a service or app
pnpm --filter @livraison/identity start:dev
pnpm --filter @livraison/web-merchant dev

Linux

corepack enable && corepack prepare pnpm@9.12.3 --activate
git clone <repo> ~/livraison && cd ~/livraison
pnpm install
docker compose up -d postgres redis
pnpm -r exec prisma generate || true
pnpm --filter @livraison/shipment start:dev

macOS

brew install node@20 pnpm
git clone <repo> ~/livraison && cd ~/livraison
pnpm install
docker compose up -d postgres redis
pnpm --filter @livraison/web-driver dev   # PWA on :3102

Verify the toolchain

pnpm turbo run lint typecheck test build
Each service has its own .env.example; each app too. Copy to .env and adjust ports/URLs for local runs.

6. Docker Deployment

Full stack with Docker Compose

cp .env.docker.example .env      # set POSTGRES_PASSWORD, JWT secrets, tenant slugs
docker compose up --build        # Postgres, Redis, 3 services, 3 apps

Services run prisma migrate deploy on start, then boot. Endpoints:

URLComponent
http://localhost:3100Merchant portal
http://localhost:3101Admin portal
http://localhost:3102Driver PWA
http://localhost:3001/api/v1/health/readyidentity health
http://localhost:3002/api/v1/health/readyshipment health
http://localhost:3003/api/v1/health/readytracking health

Building individual images

# Backend (context = service dir)
docker build -t livraison/identity:dev ./services/identity

# Web app (context = repo root; needs workspace packages)
docker build -t livraison/web-merchant:dev -f apps/web-merchant/Dockerfile .
Web images use Next.js standalone output with outputFileTracingRoot so only traced dependencies ship. All images run as a non-root user with dumb-init as PID 1.

7. VPS Deployment (Ubuntu / Debian)

Single-host production deployment with Nginx + TLS, automated backups, and monitoring. Scripts live in infra/server/.

DNS

Create A records for merchant., admin., driver., and api. subdomains pointing to the host IP.

Steps

# As root on fresh Ubuntu 22.04/24.04 (Debian 12 equivalent):
git clone <repo> /opt/livraison && cd /opt/livraison

# 1. Provision: Docker, Nginx, certbot, ufw, fail2ban, SSH hardening
sudo bash infra/server/scripts/provision.sh

# 2. Configure environment
cp infra/server/env/.env.production.example /opt/livraison/.env
sudoedit /opt/livraison/.env       # DOMAIN, secrets, REGISTRY_REPO, IMAGE_TAG
chmod 600 /opt/livraison/.env

# 3. Authenticate to the image registry
echo "$GHCR_TOKEN" | docker login ghcr.io -u "$GHCR_USER" --password-stdin

# 4. Nginx reverse proxy + TLS
sudo bash infra/server/scripts/setup-nginx.sh
sudo bash infra/server/scripts/setup-ssl.sh

# 5. Deploy (pull images, migrate, roll out, health-gate)
sudo bash infra/server/scripts/deploy.sh

# 6. Backups + monitoring timers
sudo bash infra/server/scripts/setup-backups.sh
sudo bash infra/server/scripts/setup-monitoring.sh
Debian: the same scripts work; ensure the Docker apt repo uses the Debian codename. ufw/fail2ban/certbot packages are identical.

Update / rollback

sudoedit /opt/livraison/.env          # bump IMAGE_TAG
sudo bash infra/server/scripts/deploy.sh
sudo bash infra/server/scripts/rollback.sh v1.0.0

8. Cloud Deployment

The stack is portable Docker. The simplest path on any VM provider is the VPS flow (Section 7). Below are per-platform notes.

AWS

Requirements Account, a VPC, an EC2 key pair, optional RDS/ElastiCache.

  • Setup: Launch an EC2 instance (t3.large+, Ubuntu 24.04); open 22/80/443 in the security group; allocate an Elastic IP; point DNS (Route 53) at it.
  • Deploy: Follow Section 7 on the instance. For managed data, create RDS PostgreSQL 16 + ElastiCache Redis and set each service DATABASE_URL/REDIS_URL to those endpoints (then remove the postgres/redis services from the deploy compose).

Google Cloud

  • Setup: Compute Engine VM (e2-standard-2, Ubuntu); firewall rules for 80/443; reserve a static IP; Cloud DNS records.
  • Deploy: Section 7. For managed data use Cloud SQL (PostgreSQL 16) + Memorystore (Redis).

Azure

  • Setup: Ubuntu VM (Standard_D2s_v5); NSG allow 22/80/443; public IP; Azure DNS.
  • Deploy: Section 7. Managed data: Azure Database for PostgreSQL Flexible Server + Azure Cache for Redis.

DigitalOcean

  • Setup: Droplet (2vCPU/4GB+, Ubuntu); reserved IP; DO DNS records.
  • Deploy: Section 7. Optional DO Managed Databases (PostgreSQL/Redis).

Hetzner

  • Setup: Cloud server (CX32+, Ubuntu); attach a primary IP; configure DNS at your registrar.
  • Deploy: Section 7 (best price/performance for the single-host MVP).

Railway

  • Requirements: Railway project; PostgreSQL + Redis plugins.
  • Setup/Deploy: Create one service per Dockerfile (point at the repo + Dockerfile path; web apps use repo root as build context). Add the PostgreSQL and Redis plugins; map their connection strings into each service's variables. Set the web apps' *_API_URL to the internal service URLs.

Render

  • Requirements: Render account; Render PostgreSQL + Key Value (Redis).
  • Setup/Deploy: Create a Web Service per image (Docker runtime, Dockerfile path; root for web apps). Provision Render PostgreSQL (one DB per service or schemas) and Key Value; wire env vars. Health check path /api/v1/health/ready for services, /login for apps.

Fly.io

  • Requirements: flyctl; Fly Postgres; Upstash Redis.
  • Setup/Deploy: fly launch per service/app (one app each), set the Dockerfile and build context, fly secrets set for env, fly postgres create + fly redis create and attach. Use internal .flycast hostnames for inter-service *_API_URL.
On managed-data platforms, drop the bundled postgres/redis services and point each backend at the provider endpoint. Run prisma migrate deploy once per database (the service entrypoint already does this on boot).

9. Database Setup

PostgreSQL

PostgreSQL 16. Each backend owns its own database: identity, shipment, tracking (plus livraison_core for the core-db package). The compose init script creates them automatically from POSTGRES_MULTIPLE_DATABASES.

# Manual creation (managed PG without the init script):
createdb identity; createdb shipment; createdb tracking

# Apply migrations (run once per database; entrypoint also does this):
pnpm --filter @livraison/identity exec prisma migrate deploy
pnpm --filter @livraison/shipment exec prisma migrate deploy
pnpm --filter @livraison/tracking exec prisma migrate deploy

# Optional: seed reference data (permissions, roles, demo tenant/admin)
pnpm --filter @livraison/core-db run seed
Row-Level Security: the core-db layer enables forced RLS. The application must connect as a non-owner role and set SET LOCAL app.tenant_id inside each transaction. With no tenant context, queries return zero rows (fail-closed).

Redis

Redis 7, used by the identity service for sessions, access-token revocation, and idempotency keys. Set REDIS_URL. Append-only persistence is enabled in compose; for managed Redis, enable persistence/eviction per provider.

11. Monitoring

Logs

cd /opt/livraison && docker compose logs -f identity   # per-service
journalctl -t livraison-alert                          # alert history
tail -f /var/log/nginx/access.log /var/log/nginx/error.log

Docker JSON logs rotate at 10 MB × 5 (configured in /etc/docker/daemon.json).

Health checks

Every service exposes /api/v1/health/live and /api/v1/health/ready (the latter probes the DB). A systemd timer (livraison-health.timer, every 2 min) probes all components and restarts an unhealthy container once, then alerts.

sudo bash infra/server/scripts/healthcheck.sh   # manual probe
systemctl list-timers 'livraison-*'             # see schedules

Metrics

Host resource monitor (livraison-monitor.timer, every 5 min) checks disk, memory, and container restart counts; alerts on breach via webhook or journal.

Roadmap: application metrics (Prometheus), distributed tracing (OpenTelemetry), and log aggregation (Loki) are recommended before scaling beyond the single host.

12. Backup & Recovery

Automated backups

Nightly (02:30 UTC) via livraison-backup.timer: pg_dump of each database, gzip, optional age encryption, retention pruning to BACKUP_RETENTION_DAYS.

sudo systemctl start livraison-backup.service   # run now
ls -lh /var/backups/livraison                   # inspect dumps

Restore

# DESTRUCTIVE: drops & recreates objects in the target DB; requires typed confirm
sudo bash infra/server/scripts/restore.sh shipment \
  /var/backups/livraison/shipment-<timestamp>.sql.gz
Verify restores. A backup is only real once test-restored. Restore into a scratch database monthly and confirm row counts and a few key queries.

15. Production Checklist

Before go-live

  • All default secrets replaced (DB, JWT) with strong random values
  • .env is chmod 600, never committed
  • DNS A records resolve for all four subdomains
  • TLS issued; certbot renew --dry-run passes
  • Migrations applied; seed run (or first admin created)
  • Staging deploy succeeded and smoke-checked
  • One backup taken and test-restored

Security

  • Firewall limited to 22/80/443; Postgres/Redis not exposed
  • SSH: root login + password auth disabled; fail2ban on
  • CI security scans green (audit, Trivy, gitleaks, CodeQL)
  • Public-tracking anti-enumeration enabled (roadmap item)
  • Secrets in a manager (Vault/KMS) for the next tier

Reliability

  • Health + monitor timers enabled
  • Backup timer enabled; retention set
  • Alert webhook configured
  • Rollback procedure tested (rollback.sh)

Observability (recommended next)

  • OpenTelemetry traces wired
  • Prometheus + Grafana dashboards
  • Centralized log aggregation
  • SLOs + error-budget alerts
